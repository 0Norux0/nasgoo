# Phase 12.2 — Storage & Permissions Guide

File-system permissions and storage layout for production. Do not use `chmod 777` — the app runs correctly with much more restrictive modes.

## Filesystem layout

Laravel splits storage into public (served via HTTP after `storage:link`) and private (never served directly). This project uses both.

| Path | Purpose | Web-accessible? |
| --- | --- | --- |
| `storage/app/` | Root — private by default | No |
| `storage/app/private/` | Vendor documents, invoices, customization proofs | No |
| `storage/app/public/` | User-facing files (product images, vendor logos) | Yes (via symlink) |
| `storage/framework/cache/` | Laravel framework cache | No |
| `storage/framework/sessions/` | File-driver sessions (if `SESSION_DRIVER=file`) | No |
| `storage/framework/views/` | Compiled Blade | No |
| `storage/logs/` | Application logs | No — MUST be blocked at nginx/apache |
| `bootstrap/cache/` | Bootstrap-time cache (config, routes, packages) | No |
| `public/build/` | Vite compiled JS/CSS | Yes |
| `public/storage` | Symlink to `storage/app/public/` | Yes — the symlink itself |

## Recommended permissions

For a system where the web server runs as `www-data`:

```bash
sudo chown -R www-data:www-data /var/www/marketplace
sudo find /var/www/marketplace -type d -exec chmod 755 {} \;
sudo find /var/www/marketplace -type f -exec chmod 644 {} \;

# storage/ + bootstrap/cache/ need write access for the web server
sudo chmod -R 775 /var/www/marketplace/storage
sudo chmod -R 775 /var/www/marketplace/bootstrap/cache
sudo chown -R www-data:www-data /var/www/marketplace/storage
sudo chown -R www-data:www-data /var/www/marketplace/bootstrap/cache

# .env — restrict readable to app user and web-server group only
sudo chmod 640 /var/www/marketplace/.env
sudo chown www-data:www-data /var/www/marketplace/.env

# The deployment script must be executable
sudo chmod 755 /var/www/marketplace/scripts/deploy-production-phase12.sh
```

**Never `chmod -R 777`**. It makes files world-writable — any user on the box, or an escaped PHP process, can rewrite them.

## `php artisan storage:link`

Creates the symlink `public/storage → ../storage/app/public`. Idempotent — safe to run repeatedly.

```bash
php artisan storage:link
# Expected: "The [public/storage] link has been connected to [storage/app/public]."
# Or:       "The [public/storage] link already exists." (if already linked)
```

The deploy script (`scripts/deploy-production-phase12.sh` step 11) runs this automatically.

Do NOT create the symlink manually with `ln -s` — Laravel's `storage:link` handles the correct relative path.

## Uploaded product images

Vendor uploads land in `storage/app/public/products/{product_id}/` (or a hashed path — check `ProductImageService` if present). Served at:

```
https://YOUR_DOMAIN/storage/products/123/image.jpg
```

If images don't appear after deploy:

1. Symlink missing: `php artisan storage:link`
2. Web server can't follow symlinks: check nginx config (`autoindex on;` is unrelated; the issue is usually `alias` vs `root`)
3. Permissions: `sudo chmod 775 storage/app/public/products` — the directory must be world-readable AND the file must be world-readable (644)

## Vendor documents (private)

License PDFs, ID photos, tax documents — stored at `storage/app/private/vendors/{vendor_id}/` or similar. Never served directly.

Access is through authenticated download routes only:

```bash
grep -rn "Storage::disk('private')\|Storage::disk('local')" app/Http/Controllers/ | head
```

Confirm the download endpoint requires:

- Auth
- Ownership check (vendor can only download their own; admin can download anyone's)

## Invoices / files

Depending on which phase implemented them, invoices may live in:

- `storage/app/private/invoices/`
- `storage/app/private/orders/{order_id}/`

Same rules as vendor documents: private disk, authenticated download only.

## Cache / view / logs

- `storage/framework/cache/` — safe to `rm -rf` (Laravel recreates); do it during deploy for a fully clean slate: `php artisan cache:clear`
- `storage/framework/views/` — same
- `storage/logs/` — DO NOT `rm -rf`; contains error logs. Rotate instead (see `PHASE_12_2_LOGGING_MONITORING_GUIDE.md`)

## `bootstrap/cache/`

Contains `config.php`, `routes-v7.php`, `packages.php`, `events.php`, `services.php`. These are generated by:

```bash
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan event:cache
```

The web server must be able to WRITE this directory during deploy (write ownership: `www-data:www-data`, mode 775). It must be able to READ during normal traffic.

## `public/build/`

Vite output — `npm run build` writes here. Deploy script step 4b handles this. Contents are static; safe to serve directly.

## Confirming private files are not publicly accessible

After deploy, from an external machine:

```bash
# Substitute a real private file path:
curl -I https://YOUR_DOMAIN/storage/vendors/1/license.pdf
# Expected: 404 or 403

# Attempt to fetch via /private/ (not a real public path):
curl -I https://YOUR_DOMAIN/private/vendors/1/license.pdf
# Expected: 404
```

Any `200` on a private file is a critical exposure.

## `.env` permissions

Already covered in the security report. Repeated here because it lives on the filesystem:

```bash
sudo chown www-data:www-data /var/www/marketplace/.env
sudo chmod 640 /var/www/marketplace/.env
```

`chmod 640` = owner (www-data) reads+writes, group (www-data) reads, others nothing. Standard.

## Storage-related deploy checklist

Before flipping DNS:

- [ ] `php artisan storage:link` succeeded (symlink exists)
- [ ] Uploaded product images visible at `https://YOUR_DOMAIN/storage/...`
- [ ] `curl -I https://YOUR_DOMAIN/storage/logs/laravel.log` returns 404
- [ ] `curl -I https://YOUR_DOMAIN/storage/app/private/` returns 404 (or however your private path is exposed — should NEVER be reachable)
- [ ] No files with `chmod 777` in the project tree: `find /var/www/marketplace -type f -perm -002 -not -path '*/node_modules/*'`
- [ ] `.env` permissions are 640 owned by app user
